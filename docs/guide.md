## AWS サービス アナロジー早見表

| 技術 | 一言アナロジー |
|------|--------------|
| ALB | 交通整理係（リクエストを振り分ける） |
| Multi-AZ | 別々のデータセンターに分散 |
| NAT Gateway | 出口専用の扉（外から入れない） |
| SSM Session Manager | キーなしで安全に入れる裏口 |
| SNS | ラジオ局・電波塔（放送して転送） |
| SQS | 受信ポスト（溜まって待っていてくれる） |
| HTTPS / ACM | 手紙を封筒に入れて送る |
| Secrets Manager | 金庫にパスワードをしまう |

| CloudTrail | 入退室の記録を残す受付台帳 |
| GuardDuty | 不審者を自動検知する防犯カメラ |

## Part 1：インフラ構築

### フェーズ1：構成図の説明（0〜5分）

「大きく分けると、**4つの層**で構成されています。」

- **インターネット** → ユーザーがアクセス
- **ALB**：交通整理係。アクセスを 2台のサーバーに均等振り分け。1台が壊れても止まらない
- **EC2 × 2台**：別々の AZ（東京の別データセンター）に配置。災害でも片方が動き続ける
- **RDS Multi-AZ**：Primary/Standby を 2 か所に配置。障害時は約 60 秒で自動切り替え
- EC2 はインターネットに直接つながず、ALB の後ろに隠れている（セキュリティ設計）

---

### フェーズ2：Terraform の説明 + init（5〜8分）

Terraform のメリット：
1. **同じ環境を何度でも再現**（開発・本番を同じ設定で構築可能）
2. **変更履歴が Git に残る**（誰が何を変えたか追跡可能）
3. **削除も一発**（`terraform destroy` で全リソースをまとめて削除）

```bash
terraform init
# 必要なプロバイダー（ライブラリ）をダウンロード
```

### フェーズ4：apply 実行中（12〜30分）

```bash

# RDS Multi-AZ の構築に 10〜15 分かかる
```

**待ち時間中の補足説明：**

| リソース | 説明 |
|---------|------|
| VPC | 「仮想の社内ネットワーク」。この中に置いたリソースは外から直接見えない |
| サブネット | パブリック（玄関）× 2 / プライベート（奥の部屋）× 2 |
| NAT Gateway | プライベート EC2 の「出口専用の扉」。外から入れないが中から出られる |
| RDS Multi-AZ | 2 か所同時構築中。構築後は常に同期、障害時は約 60 秒で自動切り替え |

### フェーズ6：セキュリティ対策の説明

詳細: [security.md](security.md)

**実装済み（現在の構成）：**
- Public/Private サブネット分離・EC2 直接公開なし
- Security Group で必要ポートのみ開放
- EBS・RDS ストレージ暗号化
- VPC Flow Logs（全通信を CloudWatch に記録）
- IAM ロール（アクセスキー不使用）
- SSM Session Manager（SSH ポート不要）

**追加提案（優先度順）：**

| 優先度 | 対策 | コスト感 |
|--------|------|---------|
| 今すぐ | HTTPS 化（ACM は無料） | 無料 |
| 今すぐ | Secrets Manager（DB パスワード管理） | 月 50 円〜 |
| 重要システム | WAF（Web 攻撃対策） | 月 600 円〜 |
| 重要システム | CloudTrail（操作証跡） | 月 300 円〜 |
| 大規模 | GuardDuty（AI 脅威検知） | 月 1,000 円〜 |

### フェーズ7：可用性確保の説明

詳細: [availability.md](availability.md)

**現構成の弱点：**
NAT Gateway が 1台のみ → 単一障害点（ここが落ちると EC2 のインターネット通信が止まる）

**改善案（優先度順）：**

| 優先度 | 対策 | 効果 |
|--------|------|------|
| 今すぐ | Auto Scaling Group | EC2 自動復旧・負荷対応 |
| 今すぐ | マルチ AZ NAT GW | 単一障害点の解消 |
| 重要システム | Route 53 フェイルオーバー | リージョン障害対策 |
| 大規模 | RDS リードレプリカ | 読み取り負荷分散 |

## Part 2：CPU 監視・メッセージング

### 概要

EC2-1 が 1分ごとに CPU 使用率を SNS に送信 → SQS が受け取り → EC2-2 が表示する **Pub/Sub パターン**の実装。

| サービス | 役割のたとえ |
|---------|------------|
| EC2-1（cron） | ラジオ局（1分ごとに放送） |
| SNS | 電波塔（受け取って転送） |
| SQS | 受信ポスト（溜まって待っていてくれる） |
| EC2-2 | リスナー（ポストを確認して表示） |

**Pub/Sub パターンのメリット：**
- 送る側と受け取る側が直接つながっていない（疎結合）
- EC2-2 が止まっていてもメッセージが SQS に溜まり、復旧後に受け取れる
- 受け取り側を将来的に増やすことも簡単

### 動作確認

```bash
# SNS・SQS の ARN/URL 確認
terraform output sns_topic_arn
terraform output sqs_queue_url

# EC2-2 で SQS ポーリング開始
./scripts/sqs_poller.sh <SQS_QUEUE_URL>

# 1分以内に EC2-1 からメッセージが届く
# 例: [2026-03-10T10:00:00Z] Instance: i-0abc123 | AZ: ap-northeast-1a | CPU: 3.2%
```
