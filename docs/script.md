## 全体タイムライン

| 時間 | フェーズ |
|------|---------|
| 0〜5分 | 構成図の説明 |
| 5〜8分 | Terraform の説明 + init |
| 8〜12分 | plan で確認 |
| 12〜30分 | apply（待ち時間に説明） |

---

## アナロジー早見表（迷ったらここを見る）

| 技術 | 一言アナロジー |
|------|--------------|
| ALB | 交通整理係（リクエストを振り分ける） |
| Multi-AZ | 別々のデータセンターに分散 |
| NAT Gateway | 出口専用の扉（外から入れない） |
| SSM Session Manager | キーなしで安全に入れる裏口 |
| SNS | ラジオ局・電波塔（放送して転送） |
| SQS | 受信ポスト（溜まって待っていてくれる） |
| HTTPS / ACM | 手紙を封筒に入れて送る |
| Secrets Manager | 金庫にパスワードをしまう |
| WAF | 入口に警備員を立てる |
| CloudTrail | 入退室の記録を残す受付台帳 |
| GuardDuty | 不審者を自動検知する防犯カメラ |

## 課題1：インフラ構築

### フェーズ1：構成図の説明（0〜5分）

> 「まず最初に、今日構築するシステムの全体像をご説明します。
> こちらの図をご覧ください。」
>
> 「大きく分けると、**4つの層**で構成されています。」
>
> 「一番上の **インターネット** から、ユーザーがアクセスしてきます。」
>
> 「次の **ALB（Application Load Balancer）** は、交通整理係のような役割です。
> アクセスが来たら、2台のサーバーに均等に振り分けます。
> 1台が壊れても、もう1台が対応し続けるので止まりません。」
>
> 「**EC2が2台**、それぞれ別の場所（AZ）に置いています。
> 東京の別々のデータセンターに分散させているイメージです。
> 地震などの災害で1か所が落ちても、もう1か所が動き続けます。」
>
> 「**RDS（データベース）** も同じ考え方で、2か所に置いています。
> 普段はメイン（Primary）だけが動いていて、
> 障害が起きたら自動で予備（Standby）に切り替わります。
> これを **Multi-AZ** と言います。」
>
> 「EC2はインターネットに直接つながず、ALBの後ろに隠れています。
> セキュリティ上、直接攻撃されないようにするためです。」

### フェーズ2：Terraform の説明 + init（5〜8分）

> 「今回は **Terraform** というツールで構築します。」
>
> 「普通はAWSの画面をポチポチ操作して作るんですが、
> Terraformを使うと**設計書をコードで書いて、一発で全部作れます**。」
>
> 「メリットは3つです。
> 1つ目、**同じ環境を何度でも再現できます**。開発・本番を同じ設定で作れます。
> 2つ目、**変更履歴がGitに残ります**。誰が何を変えたか追跡できます。
> 3つ目、**削除も一発です**。`terraform destroy` で全部まとめて消せます。」
>
> 「では初期化コマンドを実行します。」

```bash
terraform init
```

> 「これはTerraformに必要なライブラリをダウンロードしています。
> 料理で言うと、調理道具を棚から出してくる作業です。」

### フェーズ4：apply 実行中（12〜30分）

> 「では実際に構築します。」

```bash
terraform apply
```

> 「`yes` を入力して実行します。
> RDSの構築に10〜15分かかるので、その間に各リソースの説明をします。」

**待ち時間中の説明（順番に）：**

> 「今、VPCが作られています。
> VPCは『仮想の社内ネットワーク』みたいなものです。
> この中に置いたリソースは、外から直接見えません。」

> 「次にサブネットが4つ作られます。
> パブリックが2つ、プライベートが2つ。
> パブリックは玄関、プライベートは奥の部屋、というイメージです。」

> 「NAT Gatewayは、プライベートのEC2がインターネットに出るための**出口専用の扉**です。
> 外から入ってくることはできませんが、中から外に出ることはできます。
> ソフトウェアのアップデートなどに使います。」

> 「RDSが一番時間がかかります。
> Multi-AZなので、2か所同時に構築しているためです。
> 構築後は常に同期されていて、障害時は約60秒で自動切り替えします。」

### フェーズ6：セキュリティ対策の提案（資料説明）

> 「今構築した構成でも基本的な対策は入っていますが、
> 本番運用ではさらに強化できる点があります。資料にまとめました。」

**実装済みを先に伝える：**

> 「現在の構成で実装済みのものをまず整理します。
> EC2・RDSはインターネットから直接見えないネットワーク設計、
> セキュリティグループで必要なポートのみ開放、
> ストレージの暗号化、VPC Flow Logsによる通信記録、
> IAMロールによるキーレス認証、これらはすでに対応済みです。」

**追加提案（優先度順）：**

> 「その上で、追加で3点ご提案します。」
>
> 「まず **HTTPS化** です。
> AWSの ACM という無料の証明書サービスを使えば、
> 通信を暗号化できます。コストゼロで導入できます。」
>
> 「次に **Secrets Manager** です。
> 今はDBパスワードを設定ファイルに書いていますが、
> 金庫に入れるイメージで、コードから完全に切り離せます。
> 自動ローテーション機能もあります。」
>
> 「**WAF** は入口に警備員を立てるイメージです。
> SQLインジェクションなどのWeb攻撃を自動でブロックします。
> 重要なシステムになってきたタイミングで導入をお勧めします。」

**優先順位まとめ：**

| 優先度 | 対策 | 理由 | コスト感 |
|--------|------|------|---------|
| 今すぐ | HTTPS化（ACM） | 無料・効果大 | 無料 |
| 今すぐ | Secrets Manager | パスワード漏洩リスク排除 | 月50円〜 |
| 重要システム | WAF | Web攻撃対策 | 月600円〜 |
| 重要システム | CloudTrail | 証跡・監査対応 | 月300円〜 |
| 大規模 | GuardDuty | AI脅威検知 | 月1,000円〜 |

> 「全部いっぺんにやる必要はありません。
> システムの重要度に応じて段階的に追加するのが現実的です。」

### フェーズ7：可用性確保の提案（資料説明）

> 「続いて可用性の観点でご説明します。」

**現構成の弱点を正直に言う：**

> 「現在の構成で1点、単一障害点があります。
> NAT Gatewayが1台だけなので、ここが落ちると
> EC2からのインターネット通信が止まります。」

**改善案（優先度順）：**

> 「改善案を優先度順にご説明します。」
>
> 「**Auto Scaling Group** は、EC2の台数を負荷に応じて自動で増減させます。
> 障害で落ちたEC2を自動で新しいものに置き換える機能もあります。」
>
> 「**マルチAZ NAT Gateway** は、各AZにNAT Gatewayを1台ずつ置くことで
> 先ほどの単一障害点を解消します。」
>
> 「**Route 53フェイルオーバー** はリージョン全体の障害に備える対策です。
> 東京リージョンが落ちたら、大阪リージョンに自動で切り替えます。」

**優先順位まとめ：**

| 優先度 | 対策 | 理由 |
|--------|------|------|
| 今すぐ | Auto Scaling Group | 自動復旧・コスト最適化 |
| 今すぐ | マルチAZ NAT GW | 単一障害点の解消 |
| 重要システム | Route 53フェイルオーバー | リージョン障害対策 |
| 重要システム | RDSリードレプリカ | 読み取り負荷分散 |
| 大規模 | マルチリージョンDR | 最大限の可用性 |

> 「どの対策を優先するかは、予算・システムの重要度・想定するリスクによって変わります。
> 要件に合わせてご提案できます。」

## 課題2：CPU監視・メッセージング

### 導入（40〜42分）

> 「課題2では、サーバーのCPU使用率を
> 自動で監視・通知する仕組みを作ります。」
>
> 「登場するサービスは2つです。
> **SNS** と **SQS** というAWSのメッセージングサービスです。」
>
> 「イメージはこうです。
> EC2-1が**ラジオ局**で、1分ごとにCPU使用率を放送します。
> SNSが**電波塔**で、その放送を受け取って転送します。
> SQSが**受信ポスト**で、メッセージを溜めておきます。
> EC2-2が**リスナー**で、ポストを定期確認して内容を表示します。」
>
> 「流れをまとめると4ステップです。
>
> ① EC2-1のcronが1分ごとに自動起動
> ② CPU使用率をSNSに送信（Publish）
> ③ SNSがSQSにメッセージを転送
> ④ EC2-2がSQSを確認してCLIに表示
>
> これを **Pub/Subパターン** と言います。
> 送る側と受け取る側が直接つながっていないので、
> 将来的に受け取り側を増やすことも簡単にできます。」

---

### SNS/SQS の確認（42〜45分）

> 「SNSとSQSは、先ほどの terraform apply で一緒に作成済みです。
> 確認してみます。」

```bash
terraform output sns_topic_arn
terraform output sqs_queue_url
```

> 「これがSNSのトピックARNです。
> ARNはAWS上のリソースの住所のようなものです。
> EC2-1はここにメッセージを送ります。」

---

### EC2-1 の cron 確認（45〜47分）

*（SSM Session Manager で EC2-1 に接続）*

> 「EC2-1に接続します。
> SSHキー不要で接続できるのがSSM Session Managerの便利なところです。」

```bash
cat /etc/cron.d/cpu_monitor
```

> 「`* * * * *` というのが1分ごとを意味します。」

```bash
cat /usr/local/bin/cpu_monitor.sh
```

> 「スクリプトの中身は3つのことをしています。
> 1つ目、インスタンスIDとAZを取得。
> 2つ目、CPU使用率を計算。
> 3つ目、JSON形式でSNSに送信。」

```bash
cat /var/log/cpu_monitor.log
```

> 「ログを見ると、すでに送信が始まっています。」

---

### EC2-2 でポーリングを実演（47〜50分）

*（SSM Session Manager で EC2-2 に接続）*

> 「次にEC2-2でSQSからメッセージを受け取ります。」

```bash
./sqs_poller.sh <SQS_QUEUE_URL>
```

> 「スクリプトを起動しました。
> SQSを20秒ごとにチェックしています。
> 1分以内にEC2-1からメッセージが届きます。」

*（メッセージ表示後）*

> 「届きました。」

```
[2026-03-10T10:00:00Z] Instance: i-0abc123 | AZ: ap-northeast-1a | CPU: 3.2%
```

### なぜこの構成なのか（締め）

> 「最後に、なぜSNSとSQSを使ったかをご説明します。
>
> EC2-1からEC2-2に直接送ることもできます。
> でもそうすると、EC2-2が止まっていたらメッセージが消えてしまいます。
>
> SQSを間に挟むと、EC2-2が止まっていても
> メッセージがSQSに溜まって待っていてくれます。
> 復旧後に受け取ることができます。
>
> またSNSを使うことで、将来的にSQSだけでなく
> メールやLambdaにも同時に送れます。拡張が簡単な設計です。」
