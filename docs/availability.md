# 可用性確保のための追加構成

## 現在の構成の可用性レベル

| コンポーネント | 現在の構成 | 可用性 |
|--------------|-----------|--------|
| ALB | マルチ AZ（自動） | ◎ AWS 管理 |
| EC2 × 2 | 2 AZ に分散 | ○ 片系障害に対応 |
| RDS Multi-AZ | スタンバイ自動切替 | ◎ 自動フェイルオーバー |
| NAT Gateway | シングル（1 AZ） | △ 単一障害点あり |

---

## 追加で取るべき可用性向上構成

### 1. Auto Scaling Group（EC2 の自動スケーリング）

**目的**: トラフィック増減に応じて EC2 台数を自動調整

```
現在: EC2 × 2 台（固定）
改善: 最小 2 台 / 最大 6 台（負荷に応じて自動増減）
```

**メリット**:
- 急激なアクセス増加にも自動対応
- 障害が発生した EC2 を自動で置き換え（自己修復）
- 閑散時はコスト削減

**Terraform 実装イメージ**:
```hcl
resource "aws_autoscaling_group" "web" {
  min_size         = 2
  max_size         = 6
  desired_capacity = 2
  # スケーリングポリシー: CPU 使用率 70% 超で台数を増やす
}
```

---

### 2. マルチ AZ NAT Gateway

**目的**: NAT Gateway の単一障害点を排除

```
現在: NAT Gateway × 1 台（1a AZ のみ）
改善: NAT Gateway × 2 台（1a / 1c それぞれに配置）
```

**理由**: 現在の構成では 1a AZ の NAT Gateway に障害が発生すると、
1c AZ の EC2 からのインターネット通信（yum・AWS API）が止まる

---

### 3. Amazon Route 53 ヘルスチェック + フェイルオーバー

**目的**: ALB 自体に障害が発生した場合の切り替え

```
通常: Route 53 → Primary ALB（東京リージョン）
障害: Route 53 → Failover ALB（大阪リージョン）
```

**DR（Disaster Recovery）構成例**:

| 方式 | RTO | RPO | コスト |
|------|-----|-----|--------|
| Pilot Light | 数時間 | 数分 | 低 |
| Warm Standby | 数分〜数十分 | 数分 | 中 |
| Multi-Site Active-Active | ほぼゼロ | ほぼゼロ | 高 |

---

### 4. RDS リードレプリカ

**目的**: 読み取りクエリを分散させてパフォーマンスと可用性を向上

```
現在: RDS Multi-AZ（書き込み/読み取り 両方をプライマリに送る）
改善: プライマリ（書き込み）+ リードレプリカ（読み取り専用）
```

**メリット**:
- 読み取り負荷をリードレプリカに分散
- プライマリ障害時のフェイルオーバー先候補になる
- 別リージョンへのリードレプリカでリージョン障害にも対応可能

---

### 5. ElastiCache（セッション/キャッシュ）

**目的**: アプリケーションのセッション管理と応答速度改善

```
現在: EC2 がセッションをローカルに保持（EC2 障害でセッション消失）
改善: ElastiCache（Redis）でセッションを共有（どの EC2 でも継続）
```

**Redis クラスターモード（Multi-AZ）**: マスター + レプリカで冗長化

---

### 6. S3 + CloudFront（静的コンテンツの配信）

**目的**: 画像・CSS・JS などの静的ファイルを EC2 から切り離す

```
現在: 静的ファイルも EC2 が配信（EC2 負荷増大）
改善: S3 に静的ファイルを配置 → CloudFront（CDN）が世界中に配信
```

**メリット**:
- EC2 の負荷軽減 → 可用性向上
- 世界 400+ エッジロケーションからキャッシュ配信
- S3 の 99.999999999%（11 ナイン）の耐久性

---

### 可用性向上の優先順位まとめ

```
すぐ実装すべき（コスト効果が高い）
├── Auto Scaling Group
└── マルチ AZ NAT Gateway

重要システムで実装
├── Route 53 フェイルオーバー
└── RDS リードレプリカ

大規模サービスで実装
├── マルチリージョン DR
├── ElastiCache
└── CloudFront + S3
```
